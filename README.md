# 뉴스 검색 비교 & 저장 앱

Google News RSS 기반의 사실 데이터 수집과 Gemini를 활용한 자연어 요약, 그리고 네이버 검색 API 기반의 한국 언론사 뉴스 검색을 결합한 Streamlit 웹 애플리케이션입니다. 동일 키워드의 두 검색 결과를 나란히 비교하고, 모든 결과를 Supabase 클라우드 데이터베이스에 자동 저장합니다.

## 핵심 설계 — URL 환각 방지

LLM(Gemini) 기반 검색을 직접 사용할 경우 모델이 그럴듯한 도메인 패턴으로 **존재하지 않는 URL을 생성하는 환각(hallucination) 문제**가 빈번하게 발생합니다. 본 앱은 이 문제를 책임 분리(Separation of Concerns) 설계로 해결했습니다.

| 책임 | 담당 컴포넌트 | 이유 |
|---|---|---|
| 실제 기사 URL과 메타데이터 수집 | Google News RSS | 검색 엔진 인덱스의 결정론적 결과 — URL 환각 불가능 |
| description의 자연어 요약 가공 | Gemini API | LLM의 강점인 자연어 생성에만 활용 |
| 한국 언론사 기사 수집 | Naver News API | 공식 API 응답이라 URL 환각 없음 |

이 설계로 LLM은 절대 URL을 생성하지 않으며, 모든 링크는 검증된 외부 소스에서 직접 받아옵니다.

## 데이터베이스 스키마

검색 소스별로 결과를 분리 저장하기 위해 두 개의 테이블을 사용합니다.

### `news_history` — 구글(RSS + Gemini) 검색 결과

| 컬럼 | 타입 | 설명 |
|---|---|---|
| `id` | BIGINT | PK, 자동 증가 |
| `keyword` | TEXT | 사용자 입력 키워드 |
| `title` | TEXT | 기사 제목 (RSS 원본에서 출처명 접미사 제거) |
| `source` | TEXT | 언론사 이름 (RSS의 source 필드) |
| `news_date` | TEXT | 기사 발행일 (YYYY-MM-DD) |
| `url` | TEXT **UNIQUE** | Google News 기사 링크 — 중복 저장 방지 키 |
| `summary` | TEXT | Gemini가 생성한 3~4문장 한국어 요약 |
| `created_at` | TIMESTAMPTZ | DB 저장 시각 (자동) |

### `naver_news_history` — 네이버 검색 결과

| 컬럼 | 타입 | 설명 |
|---|---|---|
| `id` | BIGINT | PK, 자동 증가 |
| `keyword` | TEXT | 사용자 입력 키워드 |
| `title` | TEXT | HTML 태그 제거된 기사 제목 |
| `description` | TEXT | HTML 태그 제거된 본문 일부 |
| `link` | TEXT **UNIQUE** | 네이버 뉴스 링크 — 중복 저장 방지 키 |
| `original_link` | TEXT | 원본 언론사 URL |
| `pub_date` | TEXT | 발행일 (RFC 822 형식) |
| `created_at` | TIMESTAMPTZ | DB 저장 시각 (자동) |

### 테이블 생성 SQL

Supabase SQL Editor에서 아래 쿼리 실행:

```sql
CREATE TABLE news_history (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    keyword TEXT NOT NULL,
    title TEXT NOT NULL,
    source TEXT,
    news_date TEXT,
    url TEXT UNIQUE NOT NULL,
    summary TEXT,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

CREATE TABLE naver_news_history (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    keyword TEXT NOT NULL,
    title TEXT NOT NULL,
    description TEXT,
    link TEXT UNIQUE NOT NULL,
    original_link TEXT,
    pub_date TEXT,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);
```

## 코드 구조

`app.py`는 5개의 탭으로 구성되며, 각 탭의 동작 원리는 다음과 같습니다.

### 탭1. 구글 검색

검색 흐름은 두 단계로 분리됩니다.

**1단계 — 사실 데이터 수집 (`fetch_google_news_rss`)**
- Google News RSS 엔드포인트(`https://news.google.com/rss/search?q={keyword}&hl=ko&gl=KR&ceid=KR:ko`)에 HTTP 요청
- `feedparser`로 RSS 파싱
- 각 entry에서 제목, 출처(source), 발행일, URL, 원본 description 추출
- Google News가 제목 끝에 자동으로 붙이는 ` - 출처명` 접미사를 제거하여 가독성 개선

**2단계 — 자연어 요약 (`enhance_summaries_with_gemini`)**
- 5건을 한 번의 API 호출로 배치 처리하여 효율 확보
- 프롬프트에 "주어진 정보 외 어떤 사실도 추가 금지" 규칙 명시
- 응답 파싱 실패 시 RSS의 원본 description을 그대로 summary 자리에 사용 (graceful fallback)

이후 결과를 카드 UI로 표시하고, `news_history` 테이블에 자동 저장(`url` UNIQUE 제약으로 중복 방지)합니다.

### 탭2. 네이버 검색

- `search_naver_news()` 호출
- 네이버 검색 API에 `requests.get()`으로 HTTP 요청
  - 헤더: `X-Naver-Client-Id`, `X-Naver-Client-Secret`
  - 파라미터: `query`, `display=5`, `sort=date`
- 응답의 `title`, `description` 필드는 검색어 강조용 `<b>` 태그가 포함되어 있어 `clean_html()` 함수로 정제
  - `re.sub(r'<[^>]+>', '', text)` — HTML 태그 제거
  - `html.unescape(text)` — `&quot;`, `&amp;` 등 엔티티를 일반 문자로 변환
- `save_naver_news()`로 `naver_news_history` 테이블에 자동 저장 (`link` UNIQUE 제약으로 중복 방지)

### 탭3. 비교 검색

- 동일 키워드로 두 검색 함수를 순차 호출
- `st.columns(2)`로 좌우 분할하여 결과를 나란히 표시
- 하단에 정량 비교 메트릭(건수, URL 중복 건수, 구글 출처 매체 분포)과 정성 비교 인사이트 자동 표시

### 탭4. 저장된 뉴스 보기

- 라디오 버튼으로 `news_history` / `naver_news_history` 테이블 선택
- 키워드 필터링 + `st.dataframe`으로 표 출력
- CSV 다운로드 버튼 제공 (한글 깨짐 방지: `utf-8-sig` 인코딩)

### 탭5. 통계 분석

- 두 테이블을 메모리에서 병합한 후 다음 차트 생성:
  - 소스별(구글/네이버) 누적 건수
  - 키워드별 누적 검색 건수 (Top 10)
  - 일자별 저장 건수 (소스별 스택)
- 상단에 4개 메트릭 카드(구글/네이버 누적, 키워드 종류, 총 건수)

### 주요 헬퍼 함수

| 함수 | 역할 |
|---|---|
| `init_supabase()` | `@st.cache_resource`로 캐싱된 Supabase 클라이언트 |
| `clean_html(text)` | HTML 태그 + 엔티티 제거 |
| `extract_source_from_entry(entry)` | RSS entry에서 출처를 안전하게 추출 (구조 변동 대비) |
| `extract_date_from_entry(entry)` | RSS entry의 `published_parsed`를 YYYY-MM-DD로 변환 |
| `fetch_google_news_rss(keyword, max_items)` | Google News RSS에서 메타데이터 + URL 수집 |
| `enhance_summaries_with_gemini(news_list)` | RSS description을 Gemini로 자연어 요약 가공 (배치 처리) |
| `search_google_news(keyword)` | 위 두 함수를 조합한 통합 인터페이스 |
| `search_naver_news(keyword)` | 네이버 검색 API 호출 |
| `save_google_news(kw, list)` | `news_history`에 INSERT (중복 무시) |
| `save_naver_news(kw, list)` | `naver_news_history`에 INSERT (중복 무시) |

### Session State

다운로드 버튼 클릭이나 다른 탭 이동 시에도 검색 결과가 사라지지 않도록 `st.session_state`에 다음 키를 보관:
- `google_results`, `naver_results`, `compare_google`, `compare_naver`

## 구글(RSS + Gemini) vs 네이버 검색 방식 비교

| 측면 | 구글 (RSS + Gemini) | 네이버 (News API) |
|---|---|---|
| 검색 방식 | Google 뉴스 인덱스 결과를 RSS로 수신 | 네이버 검색 엔진 (키워드 매칭) |
| URL 신뢰성 | 결정론적 (RSS는 실제 검색 결과) | 결정론적 (공식 API 응답) |
| 결과 정렬 | Google News 자체 알고리즘 | 발행일 순 또는 정확도 순 |
| 결과 가공 | Gemini 자동 요약 (3~4문장) | 본문 일부(description)만 |
| 인덱싱 범위 | 전 세계 웹 (영문 매체, 블로그 포함) | 네이버 뉴스 등록 한국 매체 |
| 다국어 지원 | 가능 (`hl`, `gl`, `ceid` 파라미터) | 한국어 중심 |
| 호출 한도 | RSS 무제한 + Gemini: 분당 15회/일 1,500회 | 일 25,000회 |
| 응답 시간 | 약 2~5초 (RSS 1회 + Gemini 1회) | 1초 미만 |
| LLM 환각 위험 | 없음 (URL은 RSS에서 직접 확보) | 없음 (공식 API) |

**언제 무엇을 쓸까:**
- 빠르게 한국 언론사 최신 기사 목록이 필요하면 → 네이버
- LLM이 가공한 자연스러운 요약이 필요하면 → 구글(RSS + Gemini)
- 영문 매체나 글로벌 트렌드도 함께 봐야 하면 → 구글(RSS + Gemini)

## 환경 변수 설정

본 앱은 5개의 비밀 키를 사용하며, Streamlit Community Cloud의 Secrets 섹션에 다음 형식(TOML)으로 등록합니다.

```toml
GEMINI_API_KEY = "AIza..."
SUPABASE_URL = "https://xxxxxxxx.supabase.co"
SUPABASE_KEY = "sb_publishable_..."
NAVER_CLIENT_ID = "..."
NAVER_CLIENT_SECRET = "..."
```

### 키 발급처

| 키 | 발급처 |
|---|---|
| `GEMINI_API_KEY` | [Google AI Studio - Get API key](https://aistudio.google.com/api-keys) |
| `SUPABASE_URL` / `SUPABASE_KEY` | Supabase 프로젝트 → Connect → Project URL & Publishable key |
| `NAVER_CLIENT_ID` / `NAVER_CLIENT_SECRET` | [네이버 개발자 센터 → 애플리케이션 등록](https://developers.naver.com/apps/) (검색 API 선택) |

## 로컬 실행 (선택 사항)

```bash
# 1. 저장소 클론
git clone https://github.com/YOUR-USERNAME/YOUR-REPO.git
cd YOUR-REPO

# 2. 의존성 설치
pip install -r requirements.txt

# 3. 비밀 키 설정 — .streamlit/secrets.toml 파일 생성 후 위의 TOML 형식으로 입력
mkdir -p .streamlit
# (편집기로 .streamlit/secrets.toml 작성)

# 4. 실행
streamlit run app.py
```

> `.streamlit/secrets.toml`은 절대 GitHub에 커밋하지 마세요. `.gitignore`에 추가 권장.

## Streamlit Community Cloud 배포

1. <https://share.streamlit.io> 접속 → GitHub 계정으로 로그인
2. Create app → Deploy a public app from GitHub
3. 입력:
   - Repository: 본인 저장소
   - Branch: `main`
   - Main file path: `app.py`
4. Advanced settings → Secrets에 위 TOML 형식의 5개 키 입력
5. Deploy 클릭

## 주의 사항

- **Gemini 무료 티어**: 분당 15회 / 일 1,500회 제한. 검색 버튼을 빠르게 여러 번 누르면 429 에러 발생.
- **네이버 검색 API**: 일 25,000회 한도.
- **Google News RSS**: 명시적 한도는 없으나 과도한 요청 시 일시 차단 가능.
- **Supabase 무료 티어**: DB 500MB / Storage 1GB / 사용자 5만 명·월.
- API 키와 비밀번호는 절대 코드에 직접 포함시키지 말고 반드시 `st.secrets` 사용.
- Gemini가 생성한 요약은 RSS의 description에 근거하지만, 사실 검증을 위해서는 원본 기사 링크 확인 권장.
