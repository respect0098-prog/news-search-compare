# 뉴스 검색 비교 & 저장 앱

LLM 기반 검색(Google Gemini + Search Grounding)과 키워드 기반 검색(네이버 뉴스 API)을 동시에 활용하여 동일 키워드의 검색 결과를 비교하고, 모든 결과를 Supabase 클라우드 데이터베이스에 자동 저장하는 Streamlit 웹 애플리케이션입니다.

비즈니스머신러닝 W9-10 실습과제2 결과물입니다.

## 데이터베이스 스키마

검색 소스별로 결과를 분리 저장하기 위해 두 개의 테이블을 사용합니다.

### `news_history` — 구글(Gemini) 검색 결과

| 컬럼 | 타입 | 설명 |
|---|---|---|
| `id` | BIGINT | PK, 자동 증가 |
| `keyword` | TEXT | 사용자 입력 키워드 |
| `title` | TEXT | 기사 제목 |
| `source` | TEXT | 언론사 이름 (Gemini가 본문에서 식별) |
| `news_date` | TEXT | 기사 발행일 (YYYY-MM-DD) |
| `url` | TEXT **UNIQUE** | 기사 원본 URL — 중복 저장 방지 키 |
| `summary` | TEXT | LLM이 생성한 3~4문장 요약 |
| `created_at` | TIMESTAMPTZ | DB 저장 시각 (자동) |

### `naver_news_history` — 네이버 검색 결과

| 컬럼 | 타입 | 설명 |
|---|---|---|
| `id` | BIGINT | PK, 자동 증가 |
| `keyword` | TEXT | 사용자 입력 키워드 |
| `title` | TEXT | HTML 태그 제거된 기사 제목 |
| `description` | TEXT | HTML 태그 제거된 본문 일부 |
| `link` | TEXT **UNIQUE** | 네이버 뉴스 링크 — 중복 저장 방지 키 |
| `original_link` | TEXT | 원본 언론사 URL |
| `pub_date` | TEXT | 발행일 (RFC 822 형식) |
| `created_at` | TIMESTAMPTZ | DB 저장 시각 (자동) |

### 테이블 생성 SQL

Supabase SQL Editor에서 아래 쿼리 실행:

```sql
CREATE TABLE news_history (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    keyword TEXT NOT NULL,
    title TEXT NOT NULL,
    source TEXT,
    news_date TEXT,
    url TEXT UNIQUE NOT NULL,
    summary TEXT,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

CREATE TABLE naver_news_history (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    keyword TEXT NOT NULL,
    title TEXT NOT NULL,
    description TEXT,
    link TEXT UNIQUE NOT NULL,
    original_link TEXT,
    pub_date TEXT,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);
```

## 코드 구조

`app.py`는 5개의 탭으로 구성되며, 각 탭의 동작 원리는 다음과 같습니다.

### 탭1. 구글 검색
- `search_google_news()` 호출
- `gemini-2.5-flash-lite` 모델에 **Google Search Grounding** 도구를 활성화한 채로 프롬프트 전송
- LLM이 검색 → 요약 → JSON 포맷팅까지 한 번에 수행
- Search Grounding과 강제 JSON(`response_mime_type`) 기능은 동시 사용 불가하므로, 프롬프트에 JSON 출력을 강하게 명시한 뒤 정규표현식으로 JSON 배열만 추출
- `save_google_news()`로 `news_history` 테이블에 자동 저장 (`url` UNIQUE 제약으로 중복 방지)

### 탭2. 네이버 검색
- `search_naver_news()` 호출
- 네이버 검색 API에 `requests.get()`으로 HTTP 요청
  - 헤더: `X-Naver-Client-Id`, `X-Naver-Client-Secret`
  - 파라미터: `query`, `display=5`, `sort=date`
- 응답의 `title`, `description` 필드는 검색어 강조용 `<b>` 태그가 포함되어 있어 `clean_html()` 함수로 정제
  - `re.sub(r'<[^>]+>', '', text)` — HTML 태그 제거
  - `html.unescape(text)` — `&quot;`, `&amp;` 같은 엔티티를 일반 문자로 변환
- `save_naver_news()`로 `naver_news_history` 테이블에 자동 저장 (`link` UNIQUE 제약으로 중복 방지)

### 탭3. 비교 검색
- 동일 키워드로 두 검색 함수를 순차 호출
- `st.columns(2)`로 좌우 분할하여 결과를 나란히 표시
- 하단에 정량 비교 메트릭(건수, URL 중복 건수)과 정성 비교 인사이트 자동 표시

### 탭4. 저장된 뉴스 보기
- 라디오 버튼으로 `news_history` / `naver_news_history` 테이블 선택
- 키워드 필터링 + `st.dataframe`으로 표 출력
- CSV 다운로드 버튼 제공 (한글 깨짐 방지: `utf-8-sig` 인코딩)

### 탭5. 통계 분석
- 두 테이블을 메모리에서 병합한 후 다음 차트 생성:
  - 소스별(구글/네이버) 누적 건수
  - 키워드별 누적 검색 건수 (Top 10)
  - 일자별 저장 건수 (소스별 스택)
- 상단에 4개 메트릭 카드(구글/네이버 누적, 키워드 종류, 총 건수)
